# Build and data pipeline

The extension ships with several precomputed artifacts that are committed to the
repo. You only need to regenerate them when you change the solver, the equity
model, the feature encoding, or the training data. A normal `npm run build` does
not run any of these steps; it just bundles the committed files.

All offline commands need Node and npm on PATH. On this machine they live at
`/opt/homebrew/bin`, so prefix commands with `PATH=/opt/homebrew/bin:$PATH`.

## Stages

The pipeline runs in dependency order. Each stage writes artifacts that the next
stage (or the live build) reads.

| Stage | Command | Reads | Writes |
| --- | --- | --- | --- |
| 1. Equity matrix | `npm run solve:equity` | hand evaluator | `src/solver/preflop/equity-matrix.json` |
| 2. Preflop solve | `npm run solve:preflop` | equity matrix | `src/core/ranges/headsup-solved.ts` |
| 3. Prep tensors | `npm run ml:prep` | PokerBench CSVs in `ml/data/` | `ml/data/*.f32`, `*.u8`, `*_shape.json` |
| 4. Train net | `npm run ml:train` | prepped tensors | `src/core/ml/model.ts` |
| 5. Fixtures + parity | `npm run ml:fixtures` | `model.ts`, held-out CSV | `tests/fixtures/*.json` |

### 1. solve:equity

`src/solver/preflop/build-equity.ts` computes the 169x169 all-in equity matrix
(every starting-hand category versus every other) by Monte Carlo over random
boards, and caches it to `src/solver/preflop/equity-matrix.json`. The number of
boards per pair is tunable with `EQ_BOARDS`. The script prints spot checks
against known values (for example AA vs KK around 82.4 percent) so you can tell
at a glance that the result is sane.

### 2. solve:preflop

`src/solver/preflop/run-solve.ts` loads the equity matrix, runs CFR+ over the
heads-up preflop tree at 100bb (and shallower depths for short-stack variants),
reports exploitability, and emits the solved charts to
`src/core/ranges/headsup-solved.ts`. Iterations are tunable with `PF_ITERS`. This
file is what `gto-advisor.ts` consults first for heads-up spots.

### 3. ml:prep

`ml/prep.ts` streams the PokerBench postflop CSVs (kept in `ml/data/`, which is
gitignored), parses each row into the same normalized `Spot` the live engine
builds, runs the shared `encodeSpot`, and writes flat little-endian binary
tensors for train and test plus their shape JSON. Using the shared encoder is
what guarantees train and inference features are byte-for-byte aligned.

### 4. ml:train

`ml/train.py` trains a 2-hidden-layer ReLU MLP in pure NumPy (no torch, no
sklearn) with Adam, mini-batches, and early stopping. Accuracy is measured on
legal-masked argmax. It exports the weights and the feature standardization as
base64 Float32 into `src/core/ml/model.ts`, which the extension imports directly.

### 5. ml:fixtures

This is a convenience target that runs four steps in sequence:

- `ml/build-fixtures.ts` samples the held-out test CSV into
  `tests/fixtures/postflop-holdout.json` (stores the `Spot` and label, not
  pre-encoded features, so the test exercises the real committed encoder).
- `ml/build-parity-inputs.ts` writes `tests/fixtures/parity-inputs.json`.
- `ml/export-weights.ts` decodes the committed `model.ts` weights into
  `ml/_weights.json` (a gitignored build artifact, not the source of truth).
- `ml/parity-gen.py` recomputes reference logits in NumPy from those same
  weights and writes `tests/fixtures/parity-vectors.json`.

The parity fixtures let `ml-parity.test.ts` confirm the pure-TS forward pass in
`policy.ts` matches the trained NumPy model numerically.

## What gets committed

These generated files are tracked in git and ship with the extension:

- `src/core/ml/model.ts` (policy net weights)
- `src/core/ranges/headsup-solved.ts` (solved preflop charts)
- `src/solver/preflop/equity-matrix.json` (all-in equity matrix)
- `tests/fixtures/*.json` (accuracy and parity fixtures)

These are NOT committed (gitignored build inputs and artifacts):

- `ml/data/` (the raw PokerBench CSVs and prepped tensors)
- `ml/_weights.json` (decoded weights, regenerated by `export-weights.ts`)

## Rebuilding everything from scratch

With the PokerBench CSVs present in `ml/data/`:

```sh
PATH=/opt/homebrew/bin:$PATH npm install
PATH=/opt/homebrew/bin:$PATH npm run solve:equity
PATH=/opt/homebrew/bin:$PATH npm run solve:preflop
PATH=/opt/homebrew/bin:$PATH npm run ml:prep
PATH=/opt/homebrew/bin:$PATH npm run ml:train
PATH=/opt/homebrew/bin:$PATH npm run ml:fixtures
PATH=/opt/homebrew/bin:$PATH npm test
PATH=/opt/homebrew/bin:$PATH npm run build
```

If you only changed the solver, stages 1 and 2 are enough. If you only changed
the network or features, stages 3 through 5 are enough. Run `npm test` after any
regeneration to confirm the fixtures still match, then `npm run build` to bundle.
